# Product Discovery Knowledge

Product discovery reduces the risk of building the wrong thing by testing problem understanding before scaling implementation.

## Problem Framing

| Concern | Guidance |
|---------|----------|
| User segment | Identify who the change is for and who it is not for |
| Problem | Describe the user pain in observable terms |
| Outcome | Define what better looks like for the user and for the business |
| Constraints | Capture policy, technical, and operational boundaries early |

## Evidence and Validation

| Method | Use |
|--------|-----|
| Interviews / support signals | Understand recurring pain and context |
| Prototype / spike | Test assumptions cheaply before full build |
| Metrics / funnel data | Confirm whether the problem is meaningful and for whom |
| Counter-evidence | Look for reasons the idea may not work |

## Anti-patterns

| Anti-pattern | Problem |
|--------------|---------|
| Starting from a solution with no problem statement | Feature drift |
| No explicit target user | Success and prioritization become ambiguous |
| Treating one anecdote as proof | Overfitting |
| Skipping disconfirming evidence | Confirmation bias |
