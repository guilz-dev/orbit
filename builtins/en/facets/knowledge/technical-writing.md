# Technical Writing Knowledge

Technical writing should let the target reader act correctly with minimal ambiguity and minimal cognitive load.

## Audience-First Writing

| Concern | Guidance |
|---------|----------|
| Audience | Decide whether the reader is an operator, developer, reviewer, or end user |
| Purpose | Clarify whether the document explains, specifies, instructs, or records |
| Context | Front-load prerequisites, assumptions, and scope |
| Actionability | Prefer concrete steps, examples, and expected results |

## Structure and Style

| Topic | Guidance |
|-------|----------|
| Headings | Use headings that help scanning, not clever phrasing |
| Lists / tables | Use them when they reduce ambiguity or comparison cost |
| Examples | Prefer realistic examples over placeholders when risk is high |
| Drift control | Keep durable facts in canonical docs; move temporary analysis elsewhere |

## Anti-patterns

| Anti-pattern | Problem |
|--------------|---------|
| Writing for “everyone” | Helps no one well |
| Hiding prerequisites in the middle | Readers fail before they start |
| Long prose with no structure for procedural content | High cognitive load |
| Copying volatile details into many places | Docs drift immediately |
