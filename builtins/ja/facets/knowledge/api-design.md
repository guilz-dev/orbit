# API設計知識

REST と GraphQL API 契約のパターン。`backend` と `existing-system` 知識と併用する。

## REST 契約

| 基準 | 判定 |
|------|------|
| 誤った status code（エラーに 200、「not found」に 200） | REJECT |
| リソースパスに動詞（`/getUser`、`/createOrder`） | REJECT（名詞 + HTTP メソッド） |
| `PUT`/`DELETE` 上の非冪等セマンティクス | REJECT |
| バージョニングなしの破壊的変更 | REJECT |
| 上限のないリストエンドポイント（ページングなし） | REJECT |

- リソースは名詞。動詞はメソッドが担う。正しい status code と一貫したエラー形状を返す。
- `GET` は safe、`PUT`/`DELETE` は冪等、作成/非冪等アクションは `POST`（リトライが重要なら idempotency key をサポート）。

## ペイロードと進化

| 基準 | 判定 |
|------|------|
| 内部エンティティ/列を直接返す | 明示的なレスポンススキーマを使う |
| 非推奨化なしのフィールド削除/リネーム | REJECT（追加変更のみ） |
| エンドポイント間で casing/命名が不統一 | 警告 |
| コレクションにページング契約（cursor/offset）がない | REJECT |

- 追加的に進化させ、削除前に非推奨化する。レスポンススキーマは明示的で安定させる。
- ページング（大きな集合は cursor/keyset を優先）、フィルタ、ソートを一貫して定義する。

## GraphQL とアンチパターン

| アンチパターン | 問題 |
|--------------|---------|
| Resolver N+1（DataLoader/バッチングなし） | DB 負荷 |
| DB スキーマを 1:1 でグラフに露出 | 内部漏洩、抽象化なし |
| 上限のないクエリ深度 / コスト制限なし | DoS リスク |
| エラー契約なしで 200 にエラーを隠す | クライアントが反応できない |
| 多数往復が必要なチャッティーなエンドポイント | レイテンシ。画面のニーズ向けに設計 |
