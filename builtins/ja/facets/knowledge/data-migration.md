# データマイグレーション知識

ゼロダウンタイムデプロイ下でも安全なスキーマ・データマイグレーション。`db-schema-design`、`indexing-query-tuning` と `existing-system` と併用する。

## Expand–Contract

実行中コードが依存するスキーマを変える安全パターン:

1. **Expand** — 新しい列/テーブル/インデックスを追加（nullable/optional、後方互換）。
2. **Backfill** — バッチで新形状を埋める。旧新両方へ書き続ける。
3. **Migrate reads/writes** — デプロイの裏でアプリを新形状へ切り替える。
4. **Contract** — どのコードも読まなくなってから旧形状を削除。

| 基準 | 判定 |
|------|------|
| コード変更と同一デプロイで列の rename/drop | REJECT（ローリングデプロイを壊す） |
| デフォルトなし NOT NULL 列を大きなテーブルへ追加 | REJECT（ロック/ブロック） |
| 1 トランザクションで巨大テーブルをバックフィル | REJECT（長いロック。バッチ化） |
| ロールバック / 後方互換ステップのないマイグレーション | REJECT |

## 安全性

| 基準 | 判定 |
|------|------|
| 検証済みバックアップ / 可逆性なしの破壊的変更 | REJECT |
| 繁忙テーブルで `CONCURRENTLY` なしのインデックス作成（ロック） | 警告 |
| 冪等でない / 再実行不能なマイグレーション | 警告 |
| データ移動後の検証ステップ（件数、チェックサム）なし | 警告 |

- マイグレーションは可逆にするか forward fix とペアにする。破壊的ステップの前にバックアップする。
- 大きなバックフィルはバッチ化する。インデックスは concurrently で作る。件数/チェックサムで検証する。

## アンチパターン

| アンチパターン | 問題 |
|--------------|---------|
| 1 リリースでの big-bang rename/drop + コード | ローリングデプロイ中のエラー |
| 稼働テーブルへの長時間ロック DDL | 障害 |
| デプロイ途中でアプリとスキーマが異なる形状を前提 | ランタイム失敗 |
| 既存行のバックフィル計画なし | NULL / 壊れた不変条件 |
