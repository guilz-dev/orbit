# インシデント対応知識

本番インシデントでは、検知・安定化・連絡・復旧・学習の順で進める。`reliability-slo` と `release-rollback` と組み合わせる。

## ライフサイクル

| フェーズ | ガイダンス |
|---------|------------|
| 検知 | 症状、影響画面、開始時刻、影響範囲を確認する |
| 封じ込め | まず拡大を止める。無効化、隔離、レート制限、ロールバックを優先する |
| 復旧 | もっとも小さく可逆な操作でサービスを戻す |
| 連絡 | 事実、影響、次の更新時刻を関係者に共有する |
| 学習 | blame-less な postmortem と follow-up を残す |

- 初動では完璧な原因特定より、安定化までの時間短縮を優先する。
- 何を変えたか、誰が判断したか、どんな効果があったかを時系列で残す。

## 重大度とオーナーシップ

| 観点 | 良い実践 |
|------|----------|
| 重大度 | ユーザー影響と事業影響に結びついた severity を使う |
| インシデントリード | 調整役は 1 人、作業者は複数でもよい |
| エスカレーション | データ損失、セキュリティ露出、長時間障害の恐れがあれば早めに上げる |
| 引き継ぎ | responder が交代しても追えるように書面の timeline を持つ |

## 証拠と復旧

| トピック | ガイダンス |
|----------|------------|
| 証拠 | ログ、トレース、メトリクス、deploy/version 情報、ユーザー報告を保全する |
| 緩和策 | feature flag off、traffic shift、rollback、queue pause など可逆な手段を優先する |
| データ安全性 | データ整合性が怪しい場合は、広い復旧より先に書き込み停止を検討する |
| 検証 | “復旧した” の判定条件を決める。error rate、latency、backlog、主要ユーザーフローなど |

## Postmortem の質

| アンチパターン | 問題 |
|----------------|------|
| 証拠収集前に root cause を決め打ちする | 誤診と再発につながる |
| 目先の bug だけ直す | 検知・運用・手順の欠陥が残る |
| follow-up に owner / due date がない | 学びがシステム改善に変わらない |
| memory だけで timeline を再構成する | 重要事実が欠落・歪曲される |
