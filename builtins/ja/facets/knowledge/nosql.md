# NoSQL知識

ドキュメント / key-value ストアのモデリング（MongoDB、DynamoDB 等）。`backend` 知識と併用する。

## アクセスパターンによるモデリング

| 基準 | 判定 |
|------|------|
| リレーショナル風スキーマ設計後にアプリ側 join | REJECT |
| アクセスパターンを知る前のスキーマ設計 | REJECT |
| ドキュメント内の上限なし配列増大 | REJECT（16MB/item 制限、hot doc） |
| 常に一緒に読むデータの正規化 | 埋め込みを優先 |

- エンティティではなくクエリ中心でモデル化する: 読み書き方法を先に決め、1 回の lookup で満たす形にドキュメント/キーを設計する。
- 一緒に読むデータは埋め込み。無制限に増えるか共有されるデータは参照。

## 整合性とキー

| 基準 | 判定 |
|------|------|
| マルチドキュメントトランザクション / 強整合をデフォルト前提 | ストアの保証を確認 |
| 低カーディナリティの partition/hash key（hot partition） | REJECT |
| 大きなコレクションへのインデックスなし scan/`find` | REJECT |
| 更新戦略なしの重複データ（ずれ） | 更新経路を定義 |

- 均等分散の partition key を選ぶ。クエリ述語にインデックスを張る。フル scan を避ける。
- 非正規化はトレードオフ: 重複データには明示的な更新/修復戦略が必要。

## アンチパターン

| アンチパターン | 問題 |
|--------------|---------|
| ドキュメントストアでのリレーショナルモデリング | アプリ側 join、N+1 |
| Hot partition / hot document | スロットリング、競合 |
| 上限なし embedded 配列 | ドキュメントサイズ制限、遅い書き込み |
| eventual consistency を strong とみなす | 古い読み取り / 更新喪失 |
