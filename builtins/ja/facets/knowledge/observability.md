# オブザーバビリティ知識

ログ、メトリクス、トレーシングのパターン。`backend`、`kubernetes-docker`、`performance` と併用する。

## ロギング

| 基準 | 判定 |
|------|------|
| サービス内の非構造化 `print`/`console.log` | REJECT（構造化ログ） |
| 秘密情報 / PII / リクエスト body 全文のログ | REJECT |
| リクエスト横断で correlation/request ID がない | REJECT |
| 誤ったログレベル（error を info、ノイズを error） | 警告 |

- correlation/trace ID 付き構造化ログ（JSON）を出し、リクエストをサービス横断で追えるようにする。
- 秘密情報や PII をログしない。境界で redact する。

## メトリクスとトレース

| 基準 | 判定 |
|------|------|
| 主要パスにメトリクスなし（レイテンシ、エラー率、スループット） | REJECT |
| 症状（SLO/レイテンシ/エラー）ではなく原因（CPU）でアラート | 警告 |
| サービス境界を跨ぐ分散トレーシングなし | 警告 |
| 高カーディナリティラベルによるメトリクス系列爆発 | REJECT |

- golden signals（レイテンシ、トラフィック、エラー、飽和）を追跡する。生リソースではなくユーザー向け症状 / SLO burn でアラートする。
- サービス間で trace context を伝播する。メトリクスラベルのカーディナリティを抑える。

## アンチパターン

| アンチパターン | 問題 |
|--------------|---------|
| ログのみのテレメトリ | 集約ビューなし、クエリ高コスト |
| アラート疲労（ノイズの多いアラート） | 本番インシデントを見逃す |
| ダッシュボード / SLO なし | 正常と異常の判別不能 |
| インシデント後にだけテレメトリ追加 | インシデント中は盲目 |
