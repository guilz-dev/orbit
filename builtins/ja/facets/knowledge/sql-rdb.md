# SQL / リレーショナルDB知識

リレーショナルデータベース（Postgres/MySQL）のスキーマ、クエリ、トランザクションのパターン。`backend` 知識と併用する。

## スキーマと整合性

| 基準 | 判定 |
|------|------|
| 「速度のため」外部キー/制約を省略 | REJECT（DB に不変条件を強制させる） |
| 金額/精密値を float で保存 | REJECT（decimal/numeric を使う） |
| 常に必須な値の nullable 列 | NOT NULL を追加 |
| ずれうる派生/重複値の保存 | 意図的な非正規化でない限り REJECT |

- 不変条件はアプリだけでなくスキーマ（FK、UNIQUE、CHECK、NOT NULL）で強制する。
- 適切な型を使う: 金額は `numeric`、時刻は `timestamptz`、ネイティブ enum/boolean。

## インデックスとクエリ

| 基準 | 判定 |
|------|------|
| セットベース join の代わりにループ内クエリ（N+1） | REJECT |
| 大きなテーブルを未インデックス列でフィルタ/join | REJECT |
| ホットパスで広いテーブルに `SELECT *` | 必要列のみ選択 |
| クエリ述語/順序と合うか未確認のインデックス追加 | 警告 |

- WHERE / JOIN / ORDER BY に使う列へインデックスを張る。`EXPLAIN` で確認する。
- 行ごとの往復よりセットベースのクエリと join を優先する。

## トランザクションとマイグレーション

| 基準 | 判定 |
|------|------|
| 複数ステップの不変条件変更がトランザクションで包まれていない | REJECT |
| ロックを長く保持する長時間トランザクション | 警告（競合） |
| マイグレーションツール外でのスキーマ変更 | REJECT |
| 後方互換ステップ/ロールバック計画のない破壊的マイグレーション | REJECT |

- バージョン管理されたマイグレーションでスキーマ変更する。ゼロダウンタイムデプロイ向けに安全に設計する（列追加 → バックフィル → 切替 → 削除）。

## アンチパターン

| アンチパターン | 問題 |
|--------------|---------|
| EAV / 「神」key-value テーブル | クエリ不能、整合性なし |
| 大きなテーブルのアプリ側 join | 遅い、メモリ重い |
| 巨大テーブルの offset ページング | 深いページが遅い。keyset を優先 |
| リレーショナルデータを JSON blob で保存 | 制約/クエリ能力を失う |
