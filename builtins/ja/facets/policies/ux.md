# UXポリシー

ユーザー体験の REJECT / APPROVE 基準。Reviewer は**ここに列挙されたものだけ**を強制する。

## 原則

| 原則 | 基準 |
|-----------|----------|
| 状態網羅 | データ連動または非同期の画面は loading、empty、error、no-permission を定義する |
| 到達可能性 | すべての画面に定義された入り口（route / menu / action）がある |
| フロー完結性 | すべてのユーザータスクに定義された開始と完了/終了がある |
| デザインシステム準拠 | 既存トークン/コンポーネントを使う。カバー済みケースに色や間隔をハードコードしない |
| アクセシビリティ | インタラクティブ要素にアクセシブル名、キーボード経路、十分なコントラストがある |
| トレーサビリティ | すべての画面と要素が要件にマップされる |

## 適用範囲

ユーザー向け UI を含む変更に適用する。UI がなければ本 policy は適用しない。

## 判定基準

| 基準 | 判定 |
|----------|---------|
| データまたは非同期があるのに loading / empty / error 状態を省略した画面 | REJECT |
| 入り口が定義されていない新規画面 | REJECT |
| 完了、終了、戻りのない行き止まりユーザーフロー | REJECT |
| デザイントークンがあるのにハードコードされた色/間隔 | REJECT |
| アクセシブル名のないインタラクティブ要素 | REJECT |
| 色だけで情報を伝える | REJECT |
| モーダル/ダイアログ開閉でフォーカスを管理しない | REJECT |
| いずれの要件にもトレースできない要素 | REJECT |
| デザイン参照を超えるエッジケース状態（loading / error / empty）の追加 | OK |
| 曖昧なデザインの合理的解釈で根拠を記録 | OK |

## レビュー手順

1. UX 知識と本ポリシーを全文開く。
2. すべての `##` 節を列挙し、`ui-design.md` と実装 UI に照合する。
3. 各違反について画面/状態と policy 基準を名指しする。
4. 意図しないギャップはブロッキングとして報告する。REJECT 基準に該当しなければ APPROVE のみ。
